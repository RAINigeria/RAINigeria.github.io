What is SLAM?

SLAM — Simultaneous Localization and Mapping — is the process by which a robot builds a map of an unknown environment while at the same time determining its own position within that map. These two tasks are deeply interdependent: to build a good map you need to know where you are, but to know where you are you need a good map.

🎯 The Core Challenge
Think of being blindfolded and dropped in an unknown room. You explore by touch, remember what you've felt, and build a mental model of the space — all while tracking your own movements. SLAM gives robots exactly this ability.

This course covers SLAM across five interconnected topics, each building on the last. We start with the simplest sensor (wheel encoders) and build up to full sensor fusion with a Kalman Filter.

Part 1
Wheel Encoders & Odometry Differential Drive Kinematics

Our Angle Convention

⚠ Critical — Read Before the Formulas
In these notes, the robot's heading angle θ is measured from the vertical (y) axis, not the horizontal. This matches compass conventions: θ = 0 means facing straight up (+y), and forward motion increases y while x stays constant.

Wheel Encoders

A wheel encoder is a sensor that counts how many times the wheel has rotated. Each count is a tick. Knowing how many ticks equal one revolution and the wheel's circumference, we convert ticks directly to distance.

Ticks → Distance Conversion
distance = ticks × (π × wheel_diameter) / ticks_per_revolution

Example: diameter = 0.065 m, 500 TPR, ticks = 1250
distance = 1250 × (π × 0.065) / 500 = 0.5105 m ≈ 51 cm

Full Derivation of the Kinematics

Step A — The Arc Geometry

When the two wheels travel different distances, the robot moves along a circular arc. Both wheels are part of the same rigid body, so they must rotate about a common center called the Instantaneous Center of Curvature (ICC).

ICC * /| / | / | R + L/2 dR→ / | / | [LW]──[Robot]──[RW] | R − L/2 (inner side)

Let R = turning radius, L = wheelbase, Δθ = rotation angle. Both wheels sweep through the same angle Δθ:

dL = (R − L/2) × Δθ     ... [equation 1]
dR = (R + L/2) × Δθ     ... [equation 2]

Step B — Deriving Δθ

Subtract equation 1 from equation 2:

dR − dL = Δθ × [(R + L/2) − (R − L/2)] = Δθ × L

Δθ = (dR − dL) / L

Intuition: if dR = dL → Δθ = 0 (straight). If dR > dL → robot turns left.

Step C — Deriving d_center

Add equations 1 and 2. Since d_center = R × Δθ:

d_center = (dR + dL) / 2

Step D — Why Use θ + Δθ/2?

During the step, the robot's heading sweeps continuously from θ to θ+Δθ. Using the start angle θ (Forward Euler) or end angle θ+Δθ (Backward Euler) both introduce error. The midpoint rule (θ + Δθ/2) uses the heading at the arc's halfway point, which can be proven to give the exact result for constant-curvature arcs:

Proof — Exact Arc Integral equals Midpoint Formula x_exact = x + R × [sin(θ+Δθ) − sin(θ)]
        = x + d_center × sin(θ + Δθ/2) ✓ (trig identity)
y_exact = y + R × [cos(θ) − cos(θ+Δθ)]
        = y + d_center × cos(θ + Δθ/2) ✓

Step E — Complete Pose Update Equations

d_center = (dR + dL) / 2
Δθ      = (dR − dL) / L

x_new = x_old + d_center × sin(θ_old + Δθ/2)
y_new = y_old + d_center × cos(θ_old + Δθ/2)
θ_new = θ_old + Δθ

Check θ=0, straight ahead: x unchanged ✓, y increases ✓

Worked Example

📐 Given
Wheel diameter = 0.07 m  |  Wheelbase L = 0.20 m  |  360 TPR
Start pose: (0, 0, 0°) — facing +y
Left encoder: 180 ticks  |  Right encoder: 270 ticks
Step 1: Ticks → Distance
dL = (180/360) × π × 0.07 = 0.5 × 0.2199 = 0.1100 m
dR = (270/360) × π × 0.07 = 0.75 × 0.2199 = 0.1649 m

Step 2: Center distance and rotation
d_center = (0.1649 + 0.1100) / 2 = 0.1375 m
Δθ = (0.1649 − 0.1100) / 0.20 = 0.2745 rad ≈ 15.7°

Step 3: Update pose
midpoint heading = 0 + 0.2745/2 = 0.1373 rad
x_new = 0 + 0.1375 × sin(0.1373) = 0.0188 m
y_new = 0 + 0.1375 × cos(0.1373) = 0.1362 m
θ_new = 0.2745 rad ≈ 15.7°

Odometry Drift & Why We Need More Sensors

Odometry is an integration process — small errors accumulate with every step. In practice, robots accumulate position errors of 1–5% of total distance traveled.

Arduino Code — Encoder Odometry

Arduino C++
// Differential Drive Odometry on Arduino
// Vertical-axis convention: theta=0 faces +y

#define ENC_LEFT_A   2    // Interrupt pin
#define ENC_LEFT_B   4
#define ENC_RIGHT_A  3    // Interrupt pin
#define ENC_RIGHT_B  5

const float WHEEL_DIAMETER = 0.07;  // meters
const float WHEEL_BASE     = 0.20;  // meters
const int   TICKS_PER_REV  = 360;

volatile long ticksLeft  = 0;
volatile long ticksRight = 0;
long prevTicksLeft  = 0;
long prevTicksRight = 0;

float x = 0.0, y = 0.0, theta = 0.0;

void ISR_Left()  { ticksLeft  += (digitalRead(ENC_LEFT_B)  == HIGH) ? 1 : -1; }
void ISR_Right() { ticksRight += (digitalRead(ENC_RIGHT_B) == HIGH) ? 1 : -1; }

void setup() {
  Serial.begin(115200);
  attachInterrupt(digitalPinToInterrupt(ENC_LEFT_A),  ISR_Left,  RISING);
  attachInterrupt(digitalPinToInterrupt(ENC_RIGHT_A), ISR_Right, RISING);
}

void loop() {
  delay(50);
  long dTicksL = ticksLeft  - prevTicksLeft;
  long dTicksR = ticksRight - prevTicksRight;
  prevTicksLeft  = ticksLeft;
  prevTicksRight = ticksRight;

  float distPerTick = (PI * WHEEL_DIAMETER) / TICKS_PER_REV;
  float dL = dTicksL * distPerTick;
  float dR = dTicksR * distPerTick;

  float d_center = (dR + dL) / 2.0;
  float dTheta   = (dR - dL) / WHEEL_BASE;

  // Midpoint rule — vertical axis convention
  float midTheta = theta + dTheta / 2.0;
  x     += d_center * sin(midTheta);
  y     += d_center * cos(midTheta);
  theta += dTheta;

  // Normalize to [-PI, PI]
  while (theta >  PI) theta -= 2 * PI;
  while (theta < -PI) theta += 2 * PI;

  Serial.print("x=");     Serial.print(x, 4);
  Serial.print(" y=");    Serial.print(y, 4);
  Serial.print(" deg=");  Serial.println(theta * 180.0 / PI, 2);
}
Part 2
Ackermann Steering Odometry Car-Type Robots — Design, Build & Kinematics

What is Ackermann Steering?

Ackermann steering is the geometry used in cars, trucks, and most wheeled robots at higher speeds. Instead of spinning wheels at different speeds to turn, the robot physically pivots the front wheels. The key insight: when turning, the inner and outer front wheels must each turn by a different angle so all four wheels trace concentric circles about the same center — otherwise tyres scrub sideways and odometry fails.

The Ackermann Condition cot(δ_R) − cot(δ_L) = T / L

where T = track width, L = wheelbase
δ_L = inner wheel angle (larger), δ_R = outer wheel angle (smaller)

Building the Front Steering Mechanism

1

Make the Steering Knuckles

Each knuckle has a horizontal wheel axle hole, a vertical kingpin pivot hole, and a tie-rod arm extending inward at the Ackermann correction angle: arctan(T / 2L). For T=0.16m, L=0.20m → angle ≈ 21.8°. 3D-print or cut from 3 mm aluminum.

2

Install the Kingpins

An M4 bolt passes vertically through the chassis and the top of each knuckle. The knuckle pivots around this bolt. Use a nylon washer for smooth pivoting. Tighten with two nuts — firm enough to avoid wobble, loose enough for free rotation.

3

Attach the Tie Rod

Cut an aluminum rod to ~120 mm. Screw M3 clevises onto both ends with lock nuts. Connect clevises to the tie-rod arm holes on each knuckle. The tie rod moves both wheels together when the servo pushes or pulls it.

4

Connect the Servo

Mount the servo centered between the front wheels. A 30–50 mm pushrod connects the servo horn to the tie rod. Test manually: both front wheels should turn the same direction, inner wheel turning to a larger angle than outer wheel.

5

Add the Potentiometer Feedback

Mount a 10 kΩ pot on the kingpin bolt. As the wheel pivots, the wiper rotates. Connect to Arduino A0 and calibrate: measure raw ADC value at center, full left, and full right. Map this to steering angle in degrees.

Ackermann Kinematics — Full Derivation

Using the bicycle model (four wheels simplified to two), the rear wheel is driven and the front wheel steers. With steering angle δ and rear velocity v:

Turning Radius R = L / tan(δ)

Angular velocity of robot body ω = v / R = v × tan(δ) / L

Pose Update Equations (vertical-axis convention)
d   = ticks × (π × diameter) / TPR
dθ = (v/L) × tan(δ) × dt

x_new = x + d × sin(θ + dθ/2)
y_new = y + d × cos(θ + dθ/2)
θ_new = θ + dθ

Holonomic vs. Non-Holonomic Robots

What Does "Holonomic" Mean?

A robot is holonomic if it can move in any direction at any instant, regardless of its current heading. In 2D, a robot has 3 degrees of freedom (x, y, θ). A holonomic robot can control all three independently. Both differential drive and Ackermann are non-holonomic — they cannot slide sideways without first rotating.

⚙ Non-Holonomic Constraint
ẋ × sin(θ) − ẏ × cos(θ) = 0    (no sideways velocity — the no-slip constraint)

LiDAR-Based SLAM RPLiDAR · YLiDAR · ICP · Occupancy Grids

Physics of LiDAR

LiDAR — Light Detection and Ranging — fires a laser pulse, times how long it takes to bounce back, and computes distance using the speed of light.

Time of Flight Formula distance = (c × t_flight) / 2

c = 3 × 10⁸ m/s

Example: t_flight = 10 ns = 10 × 10⁻⁹ s
distance = (3×10⁸ × 10×10⁻⁹) / 2 = 1.5 meters

A rotating 2D LiDAR sweeps 360° and produces one (angle, distance) measurement per degree — giving 360 data points per scan. The result is a polar scan describing the cross-section of the environment at the robot's height.

Occupancy Grid Maps

An occupancy grid is a 2D matrix where each cell represents a 5–10 cm square of the environment, storing a probability: 1.0 = wall, 0.0 = free, 0.5 = unknown. We use log-odds for numerically stable updates. The Bresenham line algorithm traces which cells each LiDAR beam passes through, marking them free, and marks only the endpoint as occupied.

ICP — Iterative Closest Point

ICP finds the rotation R and translation t that best aligns two point clouds (scans). It is the core of scan matching in LiDAR SLAM.

1

Initialize

Use odometry as the initial guess for the transformation. Apply it to the source scan.

2

Find Correspondences

For every source point s_i, find the nearest point in the target scan using a KD-tree (O(log N) per query).

3

Compute Optimal Transform (SVD)

Center both point sets, compute the covariance matrix H = Σ(s_i' × t_j'ᵀ), then SVD decompose: H = UΣVᵀ. Rotation: R = V×Uᵀ. Translation: t = t̄ − R×s̄.

4

Apply Transform

Apply R and t to all source points.

5

Check Convergence

Compute MSE. If MSE < ε (e.g., 0.001 m²) — stop. Otherwise go back to Step 2.

Python · ICP from Scratch
import numpy as np
from scipy.spatial import KDTree

def icp(source, target, max_iter=50, tol=1e-4, max_dist=0.5):
    """Align source point cloud to target. Returns (dx, dy, dtheta)."""
    total_dx, total_dy, total_theta = 0.0, 0.0, 0.0
    cur = source.copy()
    tree = KDTree(target)
    prev_mse = np.inf

    for i in range(max_iter):
        dists, idx = tree.query(cur)
        valid = dists < max_dist
        if valid.sum() < 10: break

        src_pts = cur[valid];   tgt_pts = target[idx[valid]]
        sc = src_pts.mean(0);   tc = tgt_pts.mean(0)
        H  = (src_pts - sc).T @ (tgt_pts - tc)

        U, _, Vt = np.linalg.svd(H)
        D = np.diag([1, 1 if np.linalg.det(Vt.T @ U.T) > 0 else -1])
        R = Vt.T @ D @ U.T
        dtheta = np.arctan2(R[1,0], R[0,0])
        t = tc - R @ sc;  dx, dy = t[0], t[1]

        cos_a, sin_a = np.cos(total_theta), np.sin(total_theta)
        total_dx    = cos_a*dx - sin_a*dy + total_dx
        total_dy    = sin_a*dx + cos_a*dy + total_dy
        total_theta += dtheta

        cur = (R @ cur.T).T + t
        mse = np.mean(dists[valid]**2)
        if abs(prev_mse - mse) < tol: break
        prev_mse = mse

    return total_dx, total_dy, total_theta

Part 4
Indoor Positioning Systems UWB · Wi-Fi Fingerprinting · Trilateration

Why Not GPS Indoors?

GPS works by receiving signals from 4+ satellites. Indoors, buildings block and scatter these signals — accuracy drops to 10–50 meters or fails entirely. Indoor Positioning Systems (IPS) use radio, optical, or magnetic signals that work reliably inside buildings.

UWB — Ultra-Wideband Positioning

UWB sends very short radio pulses (nanosecond-wide) and precisely times how long they take to travel from a known anchor to the robot's tag. This is Two-Way Ranging (TWR):

Anchor A ──── T1: send ────→ Tag B
Anchor A ←─── T3: reply ─── Tag B (T2: received, T3: reply sent)

t_flight = [(T4 − T1) − (T3 − T2)] / 2
distance = c × t_flight = 3×10⁸ × t_flight

Example: t_flight = 5 ns → distance = 1.5 m

Trilateration — Deriving Position

With 3 anchors at known positions and measured distances, the robot's position is the intersection of 3 circles:

(x − x₁)² + (y − y₁)² = d₁² ... [1]
(x − x₂)² + (y − y₂)² = d₂² ... [2]
(x − x₃)² + (y − y₃)² = d₃² ... [3]

Subtract [1] from [2] and [3] to linearize → system Ax = b

A = |2(x₂−x₁) 2(y₂−y₁)| b = |d₁²−d₂²+x₂²−x₁²+y₂²−y₁²|
    |2(x₃−x₁) 2(y₃−y₁)| |d₁²−d₃²+x₃²−x₁²+y₃²−y₁²|

With 4+ anchors (over-determined): x = (AᵀA)⁻¹ × Aᵀ × b

Wi-Fi Fingerprinting

Every Wi-Fi access point has a unique MAC address (BSSID). The received signal strength (RSSI in dBm) at each location forms a unique pattern — a fingerprint. Two phases:

PhaseWhat You DoResult
Offline SurveyWalk through the building, record RSSI from all APs at each known locationFingerprint database
Online PositioningMeasure current RSSI, find closest match in database using KNN distanceEstimated location label

Python Code — UWB Trilateration

Python · UWB Positioning
import numpy as np

def trilaterate(anchors, distances):
    """
    Least-squares UWB trilateration.
    anchors:   [[x1,y1],[x2,y2],...] — known anchor positions (m)
    distances: [d1, d2, ...]        — measured distances (m)
    Returns: [x, y] estimated robot position
    """
    anchors   = np.array(anchors)
    distances = np.array(distances)
    x1, y1    = anchors[0];  d1 = distances[0]

    A, b = [], []
    for i in range(1, len(anchors)):
        xi, yi = anchors[i];  di = distances[i]
        A.append([2*(xi-x1), 2*(yi-y1)])
        b.append(d1**2 - di**2 + xi**2 - x1**2 + yi**2 - y1**2)

    A, b = np.array(A), np.array(b)
    pos, *_ = np.linalg.lstsq(A, b, rcond=None)
    return pos

# --- Example ---
anchors = [[0,0], [6,0], [6,4], [0,4]]   # 4 anchors in a room
dists   = [3.21, 2.83, 2.11, 3.65]        # measured distances

robot_pos = trilaterate(anchors, dists)
print(f"Robot at: x={robot_pos[0]:.3f}m, y={robot_pos[1]:.3f}m")
Part 5
Kalman Filtering & Sensor Fusion KF · EKF · Optimal State Estimation

The Core Problem

Every sensor is noisy. Every motion model is imperfect. Given multiple imperfect information sources, the Kalman Filter answers: what is the single best estimate of the robot's true state?

🏆 Why Kalman is Optimal
For systems with Gaussian noise, it is mathematically proven that no other linear estimator can produce a lower mean-square error. The Kalman Filter is the theoretical best possible.

The Kalman Filter — Two-Step Loop

Every cycle of the Kalman Filter has exactly two steps, repeated forever:

Step 1: PREDICT (using the motion model)

State transition and covariance growth
x̂_{k|k-1} = A × x̂_{k-1|k-1} + B × u_k
P_{k|k-1} = A × P_{k-1|k-1} × Aᵀ + Q

A = state transition matrix (how state evolves)
Q = process noise covariance (how much we trust the motion model)
P = covariance matrix (our total uncertainty)

After the predict step, P grows — prediction becomes less certain over time because the motion model is not perfect (process noise Q is added).

Step 2: UPDATE (using sensor measurement)

Innovation ỹ_k = z_k − H × x̂_{k|k-1}

Innovation Covariance S_k = H × P_{k|k-1} × Hᵀ + R

Kalman Gain — the core decision
K_k = P_{k|k-1} × Hᵀ × S_k⁻¹

Updated State and Covariance x̂_{k|k} = x̂_{k|k-1} + K_k × ỹ_k
P_{k|k} = (I − K_k × H) × P_{k|k-1}
💡 Kalman Gain Intuition
K = 0 → completely ignore the measurement (trust prediction only)
K = 1 → completely trust the measurement (ignore prediction)
K = 0.5 → equal weight to both

K is calculated automatically based on P (prediction uncertainty) and R (sensor noise). Large P + small R → K near 1 (trust sensor more).

Fully Worked 2-Cycle Example

📐 Setup
1D robot on a line. State: [position, velocity]
dt = 0.1 s | Q = diag([0.01, 0.1]) | R = [0.25] (UWB σ = 0.5 m)
H = [1, 0] (measures position only)
Initial: x̂₀ = [0, 0], P₀ = diag([1, 0.1])
─── Cycle 1 (t = 0.1 s) ───

PREDICT (acceleration command u = 0.5 m/s²):
x̂_{1|0} = A×x̂₀ + B×u = [0.0025, 0.05]ᵀ
P_{1|0} ≈ [[1.011, 0.01],
           [0.01, 0.20 ]]

UWB measures: z = 0.08 m

UPDATE:
Innovation ỹ = 0.08 − 0.0025 = 0.0775
S = 1.011 + 0.25 = 1.261
K = [1.011, 0.01]ᵀ / 1.261 = [0.802, 0.0079]ᵀ

x̂_{1|1} = [0.0025, 0.05] + [0.802, 0.0079] × 0.0775
        = [0.0647, 0.0506]
P_{1|1}[0,0] = 0.200 (was 1.011 → −80% uncertainty!)

─── Cycle 2 (t = 0.2 s) ───

PREDICT: x̂_{2|1} = [0.0647 + 0.0506×0.1 + 0.0025, ...] ≈ [0.0723, 0.055]
P_{2|1}[0,0] ≈ 0.210

UWB measures: z = 0.15 m
K = 0.210 / (0.210 + 0.25) = 0.456 (lower than cycle 1 — more confident)
x̂_{2|2}[0] = 0.0723 + 0.456×(0.15 − 0.0723) = 0.108 m

Extended Kalman Filter (EKF)

The standard KF requires linear models. Robot kinematics involve sin and cos — which are nonlinear. The EKF linearizes nonlinear functions at each time step using Jacobian matrices (matrix of partial derivatives).

For our differential drive robot with state x = [x, y, θ]ᵀ:

Nonlinear motion function f(x, u) f = | x + d × sin(θ + Δθ/2) |
    | y + d × cos(θ + Δθ/2) |
    | θ + Δθ |

Jacobian F = ∂f/∂x (replaces A in EKF) F = | 1 0 d × cos(θ + Δθ/2) |
    | 0 1 −d × sin(θ + Δθ/2) |
    | 0 0 1 |

EKF predict: P_{k|k-1} = F × P × Fᵀ + Q

Python · Extended Kalman Filter
import numpy as np

class RobotEKF:
    """EKF for differential drive robot. State: [x, y, theta].
    Vertical-axis convention: theta=0 faces +y."""

    def __init__(self):
        self.x = np.zeros((3,1))               # [x, y, theta]
        self.P = np.diag([0.1, 0.1, 0.05])    # Initial uncertainty
        self.Q = np.diag([0.01, 0.01, 0.005]) # Process noise
        self.R = np.diag([0.05, 0.05])         # UWB measurement noise

    def predict(self, dL, dR, L):
        """Predict from encoder readings. dL, dR in meters, L = wheelbase."""
        d  = (dR + dL) / 2.0
        dt = (dR - dL) / L
        x, y, th = self.x.flatten()
        mid = th + dt/2

        # Nonlinear motion update
        self.x = np.array([[x + d*np.sin(mid)],
                            [y + d*np.cos(mid)],
                            [th + dt]])

        # Jacobian of motion model
        F = np.array([[1, 0,  d*np.cos(mid)],
                      [0, 1, -d*np.sin(mid)],
                      [0, 0,  1            ]])
        self.P = F @ self.P @ F.T + self.Q

    def update_uwb(self, ux, uy):
        """Update from UWB measurement [ux, uy] in meters."""
        H = np.array([[1,0,0],[0,1,0]])   # measures x and y
        z = np.array([[ux],[uy]])
        S = H @ self.P @ H.T + self.R
        K = self.P @ H.T @ np.linalg.inv(S)
        self.x += K @ (z - H @ self.x)
        self.P  = (np.eye(3) - K @ H) @ self.P

    def pose(self):
        return self.x.flatten()   # [x, y, theta]

# --- Usage ---
ekf = RobotEKF()
L   = 0.20   # wheelbase

# Step 1: Predict from encoder ticks
ekf.predict(dL=0.05, dR=0.06, L=L)

# Step 2: Update from UWB every N steps
ekf.update_uwb(ux=0.15, uy=0.55)

x, y, theta = ekf.pose()
print(f"EKF pose: x={x:.3f}m, y={y:.3f}m, θ={np.degrees(theta):.1f}°")
